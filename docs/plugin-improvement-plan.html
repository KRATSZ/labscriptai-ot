1. 方案摘要

核心判断：插件的技术能力已经相当完整（7 个 Skill、40+ MCP 工具、安全策略、协议库），但用户感知到的价值被「安装门槛 + 文档分散 + 入口不清晰」掩盖。改进重点不是加更多功能，而是降低首次成功路径的长度，并把 Agent 能力包装成用户能理解的「任务入口」。

建议分三期推进：

Phase 1 · 立即可感知

5 分钟上手

统一入门文档、安装自检脚本、默认任务入口、术语表。让用户第一次对话就能跑通 simulate。

Phase 2 · 实用性提升

任务导向重组

以 opentrons-experiment-run 为默认编排 Skill，MCP 工具分层展示，内置示例实验与故障排查 Runbook。

Phase 3 · 生态完善

跨平台一致

Claude / Codex / Cursor 三端 manifest 对齐、Windows 原生安装、可选中文界面文案、发布前自动化验收。

2. 现状诊断

2.1 已有优势（应保留并突出）

模块现状评价
MCP Server opentrons-lab 覆盖仿真、预检、恢复、可选视觉、实盘控制 能力强
Skills 7 个 Skill 覆盖意图审查 → 编写 → 验证 → 修复 → 运行 → 库检索 → LAN 回退 覆盖完整
安全策略 policy/workflows.md 为单一真相源；仿真门禁、实盘 opt-in、视觉仅观测 设计成熟
编排 Skill opentrons-experiment-run 已定义完整状态机 存在但未成为默认入口
Codex manifest defaultPrompt 已提供 3 个开箱任务 仅 Codex 有

2.2 主要痛点

上手门槛高

  • 安装脚本仅 npm install，未引导 Python / Opentrons 仿真环境
  • 仅 Bash 脚本，Windows 用户需 WSL 或手动配置
  • 环境变量（OPENTRONS_PYTHON 等）无安装后自动检测
  • 三端安装路径差异大，README 信息密度高但缺少「第一步」

入口不清晰

  • 7 个 Skill 面向 Agent，用户不知道先说哪句话
  • MCP 40+ 工具无分层，新手易被工具列表淹没
  • 协议库检索需手动跑 Python 脚本，未融入对话流
  • Claude Code 的 plugin.json 缺少 Codex 那样的 defaultPrompt

文档分散重复

  • README.md、AGENTS.md、policy/、各 Skill 内嵌说明存在重叠
  • 无 docs/ 目录，缺少结构化用户指南
  • 术语（deck truth、reconcile_state、live gate）无用户向解释
  • 许可与引用要求必要但多处重复，增加阅读负担

实用场景缺口

  • 无「Hello World」端到端示例（从意图到 simulate 通过）
  • 故障排查散落在 policy 和 Skill 中，无统一 Runbook
  • 成功/失败时 Agent 输出格式不统一，用户难以判断进度
  • 实盘、视觉、探测等高级能力缺少明确的「何时启用」指引卡

3. 设计原则

  1. 仿真优先不变 — 所有改进不得削弱 simulate 门禁和实盘 opt-in 安全边界。
  2. 一个入口，多条路径 — 默认走 experiment-run 编排；高级用户仍可直达单个 Skill 或 MCP 工具。
  3. 文档单一真相源 — 工作流只维护 policy/workflows.md；用户指南只维护 docs/；其余文件链接而非复制。
  4. 失败可恢复 — 每步输出统一状态块（status、next_action、blocked_reason），让用户始终知道「现在卡在哪」。
  5. 平台差异收敛 — 三端共享同一套 Skill / MCP / policy；仅在 manifest 层做平台适配。
  6. 渐进式复杂度 — 默认隐藏高级工具（视觉、probe、recovery），在用户明确需求或出错时再 surfaced。

4. 分阶段改进计划

Phase 1：5 分钟上手 P0

4.1.1 新增 docs/GETTING_STARTED.md（三端合一，Tab 式）

用一张决策树开头，帮用户 10 秒内选对路径：

克隆仓库 → 运行安装脚本 → 环境自检 → 启用插件 → 发送第一条指令 → simulate 通过

每个平台一节：Claude Code（marketplace 命令）、Codex（本地 plugin source）、Cursor（复制 .cursor/ 配置）。附「常见失败 5 条」速查。

4.1.2 安装脚本增强

文件改动
install-labscriptai-ot.sh 安装后自动调用 health_check；检测 Node、Python、opentrons 包；输出彩色诊断报告
install-labscriptai-ot.ps1（新增） Windows 原生 PowerShell 安装，等价功能
scripts/verify-setup.mjs（新增） 独立验证脚本，可被安装脚本和三端文档引用

4.1.3 默认任务入口（三端 manifest 对齐）

将 Codex 已有的 defaultPrompt 推广到 Claude Code 和 Cursor：

推荐默认指令（用户复制即用）：
1. "帮我从零开始设计一个 Flex 移液实验，先审查意图再写协议"
2. "检查这份 Opentrons 协议能否安全仿真通过"
3. "机器人现在什么状态？帮我做恢复前的安全检查"

4.1.4 术语速查卡 docs/GLOSSARY.md

用表格解释：deck truth、simulate gate、live readiness、reconcile_state、tip policy 等。Agent 和用户共用同一词汇表。

Phase 2：任务导向重组 P1

4.2.1 确立「主 Skill」路由

在根目录新增轻量 skills/labscriptai-ot-router/SKILL.md（或增强 AGENTS.md），规则极简：
  • 新实验 / 全流程 → opentrons-experiment-run
  • 只审查意图 → opentrons-experiment-intent-review
  • 只写/改代码 → opentrons-protocol-author
  • 仿真失败 → opentrons-simulation-repair
  • 找参考协议 → opentrons-protocol-library
  • 机器人网络直连 → opentrons-robot-lan

4.2.2 MCP 工具分层（描述 + 文档，不改 API）

层级工具组默认暴露
L0 入门health_check, doctor_local_runtime, simulate_protocol, parse_simulation_output, validate_labware_name是
L1 编写辅助estimate_tip_budget, inspect_labware_definition, preflight_run_setup是
L2 实盘只读robot_status, module_status, reconcile_state, live_readiness_check需机器人 IP
L3 实盘控制run_protocol, create_run, control_run, probe_wells需明确 opt-in
L4 恢复/视觉parse_error, suggest_recovery_action, vision_check, camera_*按需

实现方式：在 MCP tool description 前缀加 [L0] 标签；在 docs/MCP_TOOLS.md 生成完整参考（可从 index.js 自动提取）。

4.2.3 内置示例实验 examples/

examples/
  01-flex-serial-dilution/     # 完整：intent brief → protocol → simulate
  02-ot2-normalization/        # OT-2 场景
  03-simulation-failure-fix/     # 故意失败 + repair 演示
  README.md                    # 每个示例的「用户该说什么」

4.2.4 统一 Agent 输出模板

在 policy/output-contract.md（新增）定义每个阶段末尾必带的 JSON 块：

{
  "phase": "simulation",
  "status": "passed | failed | blocked",
  "summary": "一句话结论",
  "next_action": "建议用户/Agent 的下一步",
  "artifacts": ["protocol.py", "sim-log.txt"]
}

4.2.5 故障排查 Runbook docs/runbooks/

  • simulation-fails.md — 最常见 10 种仿真错误
  • mcp-wont-start.md — Node / 路径 / 环境变量
  • robot-connection.md — IP、LAN fallback 与 MCP 互斥说明
  • after-reboot.md — safe_next_action 流程

5. 信息架构重组

建议最终文档结构：

labscriptai-ot/
├── README.md                    # 极简入口（< 80 行）
├── AGENTS.md                    # Agent 行为契约（链接 policy，不重复工作流）
├── docs/

用户心智模型（建议对外宣传）

说清楚实验 → 生成协议 → 本地仿真 → 确认安全 → 上机执行

技术层的 7 Skill + 40 MCP 工具折叠在这 5 步背后，用户只需记住一句话：「先仿真，后上机；不确定就问机器人状态。」

8. 待讨论决策点

以下问题影响方案细节，建议先对齐再动手：

Q1：主入口放哪？

选项 A：增强现有 opentrons-experiment-run，在 AGENTS.md 声明为默认 Skill。
选项 B：新增 labscriptai-ot-router 做显式路由。
倾向：A 更简；若三端 manifest 支持「默认 Skill」字段则选 A，否则 B 更清晰。

Q2：文档语言策略？

选项 A：用户文档中英双语（docs/ + docs/zh-CN/）。
选项 B：仅中文用户文档，Skill / policy 保持英文。
选项 C：全英文，靠 Agent 实时翻译。
团队主要用户群决定此选择。

Q3：安装脚本是否捆绑 Python 环境？

轻量方案：只检测 + 打印安装指引（推荐，维护成本低）。
重量方案：脚本内建 venv + pip install opentrons（体验好但跨平台维护难）。

Q4：MCP 工具是否物理分组（多个 MCP server）？

现状：单一 opentrons-lab server。
方案：仅文档分层（推荐）vs 拆成 opentrons-sim + opentrons-live 两个 server（隔离更彻底但配置复杂）。

Q5：示例实验的仿真依赖？

示例协议是否保证「零配置即可 simulate 通过」？若需特定 labware 版本，需在示例 README 标明最低 Opentrons 版本。

Q6：Phase 1 的范围边界？

是否接受 Phase 1 只做文档和 manifest，不改 MCP 代码？这样 1–2 天可交付可感知改进；自检脚本可作为 Phase 1 唯一代码改动。
